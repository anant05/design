Initiative Lifecycle Documentation

How compliance initiatives move from idea to active enforcement
Introduction

What is a CI Compliance Initiative?

A compliance initiative is a formal requirement that teams must meet in order to merge code. Once active, the CI system automatically checks each PR and can warn or block based on compliance status.

Automated Enforcement
CI checks run on every PR, warning or blocking based on compliance state.
Manual Override
When automation isn't possible, project teams can provide evidence and manually mark projects compliant.
Gradual Rollout
Warn phase first, then blocking — giving teams time to remediate.
Key Roles
AG
Initiative Author
Creates the initiative, sets the enforcement timeline and scope, submits for CI review, and manages enforcement once active.
CI
CI Team
Reviews feasibility, builds the backend enforcement check, tests it, and deploys it to production. Controls all status transitions.
Project Teams
Receive warnings/blocks on PRs. Make the technical changes needed to become compliant or provide override evidence.

Projects

Project-Level Compliance

Once an initiative is Ready for Activation, each in-scope project is tracked individually. Compliance status can be set automatically by CI or manually overridden with evidence.

Project Statuses
CompliantCI check passes. No action needed.
Non-CompliantCheck failing. PRs may be warned/blocked.
In ProgressTeam is actively remediating.
N/AInitiative does not apply to this project.
Manual Override
When full automation isn't possible (e.g., legacy projects, special toolchains), a project owner can manually override the status by providing evidence:
✓Link to PR or commit verifying the change
✓Description of manual verification steps
✓Approval from team lead
Manual overrides are logged in the initiative's project activity trail.
PRs Blocked — counts the number of pull requests that were blocked by this initiative's CI check. This metric is visible per-initiative and per-project, and resets when a project becomes compliant.
Enforcement

Managing Active Enforcement

Warn Phase
CI adds a warning comment to non-compliant PRs. PRs are not blocked — teams are notified and can still merge.
Starts on the Warn Date
Block Phase
CI blocks non-compliant PRs from merging. Teams must remediate or obtain a manual override to proceed.
Starts on the Block Date
⏸ Pause Enforcement
Temporarily suspends all CI checks (warn & block) without changing dates. Use when enforcement needs to be paused for a deployment window or incident.
Available in Ready state. Resuming restores enforcement immediately.
🗄 Deprecate Initiative
Permanently retires the initiative. The CI check is removed from production pipelines. All project statuses are cleared. This action cannot be undone.
Available to initiative owner or CI team.
CI Team

CI Team Workflow

1
Triage Submitted Initiatives
Review the CI Review Dashboard for newly submitted initiatives. Assess the request and move to Review to indicate work has begun.
2
Conduct Feasibility Review
Schedule a meeting with the initiative author. Assess technical feasibility, effort estimate, and rollout impact. Move to Development or Rejected based on outcome.
3
Build Backend Check
Implement the CI enforcement check. Coordinate with the author on technical requirements. Log progress comments in the initiative activity log.
4
Test in Staging
Deploy the check to staging. Run validation tests. Confirm correct warn/block behavior. The initiative is fully locked at this stage to preserve test integrity.
5
Deploy to Production → Ready
Deploy the check to production. Move initiative to Ready for Activation. The author can now adjust enforcement dates and scope to begin the rollout.
